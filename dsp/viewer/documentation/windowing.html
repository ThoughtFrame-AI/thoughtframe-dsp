Windowing & Events

Windowing is the mechanism by which continuous acoustic streams are transformed into discrete, reviewable units of evidence. Rather than attempting to classify individual samples or chunks in isolation, the system reasons over time-bounded windows with explicit lifecycles.

Why Windowing Exists

Raw acoustic data is unbounded. Even per-chunk features are too granular to support meaningful interpretation or human review. Windowing exists to impose structure on time without losing continuity.

Instead of asking “what is this chunk?”, the system asks:

“Over this span of time, what happened, and how did it evolve?”

This shift is essential for real-world sensing, where events have duration, onset, decay, and internal variation.

Window Isolators

Windowing is implemented by specialized processors called WindowIsolators. These processors observe the stream of chunk-level measurements and propose transitions between window states.

A window isolator:

  • Maintains internal state across chunks
  • Defines explicit rules for window boundaries
  • Accumulates aggregate statistics over time
  • Persists completed windows as durable records

Importantly, isolators do not infer meaning. They segment time according to explicit, inspectable logic.

Window Lifecycle

Every window follows a well-defined lifecycle:

  • Start – a new window is opened at a specific time and chunk
  • Accumulate – statistics are updated as chunks arrive
  • Close – the window is finalized based on explicit criteria
  • Persist – aggregated results are written to disk

Lifecycle boundaries are explicit and deterministic. Short or spurious windows may be discarded using minimum duration rules to suppress noise.

Base Window Schema

All windows share a canonical schema, ensuring consistency across isolators and pipelines. This schema includes:

  • Unique window identifier
  • Start and end times
  • Start and end chunk indices
  • Duration in seconds and chunks
  • Number of contributing chunks
  • Aggregate acoustic statistics (mean, variance, min, max)

Individual isolators may extend this schema with additional fields, but the base lifecycle and timing fields are fixed.

From Windows to Events

A completed window represents a candidate event. At this layer, an event is not a semantic label or classification — it is a bounded region of time with measurable properties.

Events are therefore:

  • Time-bounded
  • Replayable
  • Comparable across runs
  • Independent of interpretation

This definition allows events to be reinterpreted later as new context, models, or human insight becomes available.

Multiple Windowing Strategies

Different isolators may be used simultaneously within a pipeline, each capturing different temporal phenomena. For example:

  • Fixed-duration windows for baseline statistics
  • Novelty-driven windows for anomalous activity
  • Impulse windows for transient signals
  • Contextual windows spanning elevated activity

Because windows are independent and persisted, overlapping or parallel interpretations do not conflict.

Persistence and Review

Completed windows are written immediately as structured records, typically in CSV form. This persistence layer is not optional or best-effort — it is part of the windowing contract.

Persisted windows enable:

  • Offline analysis and visualization
  • Human review and annotation
  • Cross-run comparison
  • Reprocessing without raw audio

Why Windows Matter

Windowing is the bridge between continuous sensing and structured reasoning. It allows the system to operate indefinitely while still producing discrete, trustworthy units of evidence.

By making window boundaries explicit and inspectable, the system avoids the ambiguity and irreproducibility common in end-to-end streaming models.
